# 📝 C群：イベント管理機能
## イベントとは
イベントは、クラブで開催され、クラブメンバーが参加する練習会、レッスン会、大会を指す。
### イベント構成要素
- イベント名：イベントを識別する名前。「練習会」など
- 開催日：イベントが開催される日
- 開催開始時間：イベント開始時間
- 開催終了時間：イベント終了時間
- 場所：イベントが行われる場所
- 定員：イベントに参加可能な人数
- 最低催行人数：イベントを開催するために最低限必要な人数
- 参加レベル下限：イベントに参加可能なレベル下限値(DUPR値)
- 参加レベル上限：イベントに参加可能なレベル上限値(DUPR値)
- 駐車場可否：イベント会場の駐車場の有無
- 駐車場台数：イベント会場で使える駐車場数
- 試合形式：シングルス、ダブルス
- 対戦形式：男子ダブルス、女子ダブルス、ミックスダブルス、オープン
- 単独参加可否：1人でも参加可能か、ペアでの参加のみか
- ゲスト参加可否：ゲストが参加できるか
- 費用：イベント参加に必要な費用
- ゲスト費用：ゲストが参加する場合の費用
- 申し込み開始日：イベント申込開始日
- 申し込み締め切り日：イベント申込終了日
- 抽選日：定員オーバー時の参加者を決定する抽選日
- 注記：イベントに関する注意事項など
- イベント状態：[下書き、公開、無効]
- イベント状況：[募集前、募集中、応募締め切り後、参加者確定済み、中止、終了]
- イベント作成者：イベントを作成した人(会員番号)
- イベント更新者：イベントを更新した人(会員番号)
- イベント作成日時：イベントを作成した日時
- イベント更新日時：イベントを更新した日時

### イベントに参加する人の要素
- イベント登録者
- イベント参加登録日時
- イベント参加情報更新日時
- イベント参加ペア名(member_number)

### イベント概要
イベント概要とは、イベント一覧に表示されるイベント情報。次の情報がイベント概要で表示される
- イベント名、開催日、開催開始時間、開催終了時間、場所、参加レベル下限値～上限値、定員、参加者のアイコン
### イベント詳細
イベント詳細とは、イベント情報の詳細。イベント概要で表示される情報に加え、次の情報がイベント詳細で表示される
- 最低催行人数、駐車場可否、駐車場台数、試合形式、対戦形式、単独参加可否、ゲスト参加可否、費用、ゲスト費用
　申し込み開始日、申し込み締め切り日、抽選日、注記

## ユースケース
### 一般利用者向けユースケース
1. イベントの参照と検討
- C-01：イベントの開催予定を確認する
会員は、カレンダーまたはリスト形式で表示される一覧から、
現在募集中のイベント、および今後開催予定のイベントを閲覧できます。
また、自分が過去に参加したイベントの履歴や結果も参照できます。

- C-02：イベントの詳細を確認する
会員は、興味のあるイベントを選択し、
日時、場所（施設）、費用、対象レベル、駐車場の利用可否・残数などの募集要項を確認します。
また、自分の申込状況（例：「抽選待ち」「当選」「キャンセル待ち」など）もここで確認できます。

2. 参加の申し込み（募集期間中）
- C-03：イベントに参加を申し込む（エントリー）
会員は、募集期間中にエントリーを行います。
車で来場する場合は、駐車場の利用希望も併せて登録します。
  - 単独申請：自分一人で参加を希望する。
  - ペア申請（指名）：特定の相手を指名して申し込む。相手が承諾するまで「保留」となる。
  - ペア申請（承認）：自分を指名している相手を指名し返し、正式にペアとしてエントリーを完了する。

- C-04：申し込み内容を変更・取り消す
募集締切前であれば、ペアの変更、駐車希望の修正、または申込自体の取り消しが可能です。

3. 当選確定後および当日の操作
- C-05：ペア相手の変更をリクエストする
当選確定後にペア相手が不参加となった場合、
会員は当選権利を維持したまま、新しいペア候補を指名して管理者へ承認を申請します。

- C-06：当選権利を辞退する
参加できなくなった場合、当選を辞退できます。
また、イベントには参加するが車を使わなくなった場合、駐車枠のみ辞退することもできます。

- C-07：主催者へ連絡事項を伝える
遅刻の連絡や運営への相談など、主催者へメッセージを送信し、やり取りを記録します。

### 管理者・イベント担当向けユースケース
1. イベントの起案と公開
- C-11：イベントを新規登録する
管理者は、開催内容を入力し、
  - 下書きとして保存する  
  - 即座に公開する  
のいずれかを選択できます。
また、既存イベントをコピーして新規作成することもできます。

- C-12：不要なイベントを取り消す（無効化）
応募者が一人もいないイベントに限り、一覧から非表示にして無効化（論理削除）できます。

2. イベント内容の編集
- C-13：イベント内容を編集する
募集開始前であれば、全項目の修正が可能です。
募集開始後は、参加者の混乱を避けるため、誤字脱字などの軽微な修正に限定されます。
また、施設マスタの標準値（例：駐車場台数）を、イベント単位で上書きすることもできます。

3. 状況管理と選考・調整
- C-14：募集の進行をコントロールする
管理者は、以下の操作を行います。
  - 下書き → 公開
  - 募集の早期終了
  - 開催中止の決定（応募者全員へ通知）

- C-15：参加者を確定させる
募集締切後、設定された「抽選日」または管理者の操作により抽選を開始します。
システムが仮当選を生成した後、管理者は
キャンセル待ち順位や駐車枠の割り当てを調整し、結果を一斉公開します。
また、会員からのペア変更申請（C-05）を審査し、問題なければ確定します。

4. 当日の運営と事後処理
- C-16：出席確認と完了処理
当日の来場確認と参加費の受領を記録します。
全工程が終了したら「完了」を確定し、実績データを会計システムへ連携します。

- C-17：参加者へ個別・全体連絡を送る
会員からのメモへの回答や、参加者全体への連絡事項を送信します。

### 実装・設計時の注意事項（曖昧さを排除した版）
1. 時刻と状態の複合バリデーション
システムは、
  - 状態（Status）
  - 現在時刻  
の両方を用いて可否判定を行う必要があります。

2. 編集制限
管理者が全項目を編集できるのは以下のいずれかです：
  - 状態が 下書き（draft）
  - 状態が 公開中（published） かつ 現在時刻 < 募集開始日時

募集開始後は、料金・定員・対象レベルなど、参加者に影響する項目の変更を禁止します。

3. 申請制限
一般会員が申請・変更できるのは：
  - 状態が 公開中（published）
  - 募集開始日時 ≤ 現在時刻 < 募集締切日時
募集開始日時が未設定（null）の場合は、
募集開始前として扱うものとします。

4. 抽選フェーズの隠蔽
募集締切後は「選考中（selecting）」状態とし、
管理者が「結果公開」を行うまで、会員には結果を一切表示しません。

5. 駐車枠の連動管理
駐車枠は参加枠とは独立して抽選・管理します。
駐車枠のみ辞退された場合は、次点者への繰り上げまたは管理者による再割り当てが可能な構造にします。

6. 当選枠の継承ルール
当選後のペア変更は、直接更新を禁止し、必ず「変更リクエスト」＋管理者承認のフローを通します。

7. 物理削除の禁止
イベントは物理削除せず、履歴を保持します。
  - 応募 0 件 → 無効化(disabled)。応募者が0件の場合のみ、誤作成として処理することを許容します。
  - 応募 1 件以上 → 中止(cancelled)。1件でも応募がある場合は、必ず中止扱いとし、誰が申し込んでいたかという証跡を永久に保持してください。

## イベント状態遷移図（State Machine）
### 状態一覧
| 状態	| 説明 | 
| --- | --- | 
| draft	| 下書き。管理者のみ閲覧・編集可能 | 
| published	| 公開中。募集開始前後で編集制限が変わる | 
| closed	| 下書き。管理者のみ閲覧・編集可能 | 
| selecting	| 選考（抽選）・調整中。結果は非公開 | 
| confirmed	| 結果公開済み。当選・落選が確定 | 
| finished	| 完了。実績ロック | 
| cancelled	| 中止。履歴保持・応募者へ通知 | 
| disabled	| 無効化（論理削除）。応募 0 件のときのみ |

### 遷移条件（要件定義として明確化）
draft → published
- C-14a（一般公開）
- C-11b（直接公開）

published → closed
- 募集締切日時の到達
- C-14b（早期終了）

closed → selecting
- 抽選日（lottery_date）の到来
- または C-15a（管理者が抽選開始を実行）

selecting → confirmed
- C-15b（最終確定・結果公開）

confirmed → finished
- C-16b（完了確定）

任意状態 → cancelled
- C-14c（中止）
- ※応募者へ通知必須

draft / published → disabled
- C-12b（応募 0 件のときのみ無効化）
- ※応募 1 件以上の場合は cancelled を使用

### 状態遷移の文章定義（要件定義書にそのまま使える版）
■ 募集期間中の遷移
- 会員が単独で申し込むと pending になる
- ペア申請の片側のみ成立した場合は pair_pending
- 相互指名が成立すると pending に戻り、抽選対象となる
- 募集締切前であれば、本人はいつでも cancelled に遷移できる

■ 抽選フェーズの遷移
募集締切後、pending の参加者は抽選により以下のいずれかに遷移する：
- won（当選）
- waitlist（キャンセル待ち）
- lost（落選）
pair_pending のまま締切を迎えた場合は抽選対象外となり、自動的に cancelled と同等扱い。

■ 当選後の遷移
- 当選者が辞退すると declined
- ペア相手が不参加となり、新ペア候補が管理者承認された場合、元の当選者は replaced へ遷移し、当選枠は新ペアへ移譲される

■ 当日の遷移
- 当選者が出席すると attended
- 当選者が欠席すると absent

## 駐車場枠（ParkingRequest）状態一覧
| 状態 | 説明 |
| --- | --- |
| pending |	説明 | 
| requested	| 会員が「駐車場を希望する」と申請した状態（募集期間中）。 | 
| cancelled	| 募集期間中に本人が駐車場希望を取り消した。 | 
| lost 	| 抽選で駐車場に落選した。 | 
| waitlist | 駐車場のキャンセル待ちになった。 | 
| won	| 駐車場に当選した。 | 
| declined | 当選後に本人が駐車場だけ辞退した。 | 
| reassigned | 当選者の辞退により、次点者へ繰り上げ当選した。 | 

※「参加辞退（イベント自体の辞退）」は EventParticipation 側で管理し、
駐車場側は 自動的に cancelled 扱いになる。

### 状態遷移の文章定義（要件定義書にそのまま使える版
■ 募集期間中の遷移
- 会員が駐車場を希望すると requested になる
- 募集締切前であれば、本人はいつでも cancelled に遷移できる
- 参加申請自体を取り消した場合も、駐車場希望は自動的に cancelled となる

■ 抽選フェーズの遷移
募集締切後、requested のユーザーは抽選により以下のいずれかに遷移する：
- won（駐車場当選）
- waitlist（駐車場キャンセル待ち）
- lost（駐車場落選）

■ 当選後の遷移
- 当選者が「駐車場だけ辞退」すると declined
- 辞退が発生した場合、キャンセル待ちの先頭が reassigned（繰り上げ当選） に遷移する
- 繰り上げ当選後、元の waitlist は繰り上げ順に再計算される

■ 参加辞退との関係
- イベント参加自体を辞退した場合（EventParticipation が declined などに遷移した場合）
  - → ParkingRequest は自動的に cancelled へ遷移する
  - → 駐車枠は次点者へ繰り上げ可能

## ペア（Pair）状態一覧
| 状態 | 説明 |
| --- | --- |
| forming	| ペア申請の片側のみ成立（相手承認待ち）。抽選対象外。| 
| active | ペアが正式に成立し、抽選対象となる状態。| 
| broken_by_user | 募集期間中に片方がペアを解消した状態。| 
| broken_after_win | 	当選後に片方が不参加となり、ペアが解消された状態。| 
| replaced | ペア解消後、当選枠が新しい相手に移譲され、ペアが再成立した状態。| 
| dissolved	| イベント終了後、または管理者操作により完全に解散した状態。|

### 状態遷移の文章定義（要件定義書にそのまま使える版）
■ 募集期間中の遷移
- forming（片側のみのペア申請）
  - 片方が相手を指名した時点で forming
  - 相手が承認すると active
  - 相手が承認しないまま募集締切を迎えると、ペアは成立せず、
EventParticipation は単独扱いまたは cancelled 扱いになる

- broken_by_user（募集期間中の解消）
  - どちらかがペアを解除すると broken_by_user
  - 解除後は単独申請に戻る（EventParticipation 側で pending に戻る）

■ 抽選フェーズ
- active のペアのみ抽選対象
- forming や broken_by_user は抽選対象外

■ 当選後の遷移
- broken_after_win（当選後のペア解消）
  - 当選後に片方が不参加となると broken_after_win
  - この時点で当選枠は「代表者側」に残る（EventParticipation 側のルール）

- replaced（新ペアへの枠移譲）
  - 会員が C-05（ペア変更リクエスト）を提出
  - 管理者が C-15c で承認するとbroken_after_win → replaced
  - replaced は「新しいペアが当選枠を引き継いだ」状態

■ イベント終了後
- dissolved（完全解散）
  - active / replaced / broken_after_win のいずれもイベント終了後に dissolved へ遷移
  - 履歴としては保持するが、以降の操作は不可
